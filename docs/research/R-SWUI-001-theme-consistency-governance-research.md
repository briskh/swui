# R-SWUI-001：主题与尺寸一致性专项治理研究报告

- **文档编号**：R-SWUI-001
- **状态**：implemented；原始研究范围已由 F-SWQT-0001、F-SWQT-0002 与 F-SWQT-0003 分阶段落实，最终交付复核已由 F-SWQT-0003 完成
- **记录类型**：research evidence / Feature 输入
- **日期**：2026-07-16
- **范围**：`@swui/ui-tokens`、`@swui/ui` 及其消费端接入方式

## 1. 执行摘要

### 1.1. 结论

组件库已具备语义颜色 token、Tailwind v4 `@theme inline` 桥接和大量语义化组件样式，适合作为主题与尺寸系统的唯一真源。当前主要问题不在于缺少 token，而在于公开 API、运行时行为、CSS 预留能力、尺寸规范和验证机制尚未共同构成可审计的契约。

后续 Feature 应正式交付浅色/深色主题与全局主题选择，并同时建立尺寸系统。尺寸系统以稳定的 4px 节奏、控件尺寸、容器尺度、全局响应式断点和数据密度专用阈值组成；它与颜色主题共享 token、文档、测试和发布治理，但默认不随明暗主题改变。

### 1.2. Feature 目标

1. 交付 `system | light | dark` 的全局主题偏好模型及真实可用的 `ThemeControl`。
2. 让 token、组件、运行时、文档与消费端在浅色和深色主题下表达同一能力。
3. 将现有零散的尺寸惯例收敛为可复用、可验证的尺寸 token 与组件尺寸契约。
4. 为主题、尺寸、无障碍、视觉回归和发布态消费建立合格门禁。

### 1.3. 成功定义

- 用户未设置偏好时跟随操作系统；显式选择可跨刷新和浏览器标签页同步，并优先于系统偏好。
- 每个正式主题均满足 WCAG AA、核心组件自动无障碍检查、视觉回归和发布产物消费验证。
- Button、Input、Select、图标按钮等基础交互控件共享公开且一致的尺寸变体。
- 共享组件不再依赖未登记的裸色值、裸尺寸值或主题特例；已批准的例外可审计。

## 2. 范围、边界与术语

### 2.1. 范围

- `packages/ui-tokens`：语义 token 真源、Tailwind 主题变量和主题选择器。
- `packages/ui`：主题运行时、基础组件、`ThemeControl`、尺寸变体及其文档。
- 消费端：token CSS 导入、`@source` 扫描、发布产物集成和主题初始化。
- 验证设施：组件状态故事、无障碍、视觉回归、静态规则和发布态 smoke test。

### 2.2. 明确不在范围内

- 不引入 AppShell、TopBar、路由或产品级页面网格。
- 不要求每一项页面布局属性都由组件库 token 化；消费端仍负责具体页面编排。
- 不在本 Feature 中把 CSS token 强制迁移为 DTCG JSON；仅采用其分层、引用、元数据和弃用治理思想。
- 不承诺子树级主题隔离、多品牌主题或高对比度第三套主题。

### 2.3. 术语

| 术语 | 定义 |
| --- | --- |
| 主题偏好 | 用户选择的 `system`、`light` 或 `dark`；`system` 是策略，不是第三套视觉主题。 |
| 有效主题 | 运行时最终应用的 `light` 或 `dark`。 |
| 基础 token | 色阶、尺寸等原始值；组件不直接消费。 |
| 语义 token | 如 `background`、`foreground`、`border`、`focus`、`space-panel`，表达用途而非原始数值。 |
| 组件 token | 仅在通用语义无法准确表达时使用的组件级语义。 |
| 数据密度阈值 | `data-dense` 专用显示门槛，用于无法在窄屏可靠适配的密集数据控件。 |

## 3. 当前状态与本地证据

### 3.1. 已具备的基础

| 维度 | 本地证据 | 结论 |
| --- | --- | --- |
| token 真源 | `packages/ui-tokens/src/tokens.css` 集中定义语义变量与 `@theme inline` 映射 | 可作为唯一主题/尺寸真源的落点。 |
| 组件样式 | 组件普遍使用 `bg-background`、`text-foreground`、`border-input`、`ring-ring` 等语义类 | 语义化方向正确，迁移成本可控。 |
| 尺寸惯例 | 组件与设计摘要已有 4px 节奏、常用 8/12/16/24/32px 间距、默认 `h-10`（40px）控件 | 已有稳定基线，但尚未形成公开 token 契约。 |
| 数据密度处理 | `WideScreenGate` 以 `max-data-dense` / `min-data-dense` gate 密集表格 | 已有专用策略，不应被误作全局断点。 |
| 消费端接入 | 文档与 `examples/ui-consumer` 已证明 token CSS 导入和 `@source` 扫描路径 | 有可用接入起点，尚无发布态覆盖。 |

### 3.2. 历史立项时的实现与目标契约差异

本表记录的是立项时的差异，不是当前状态。当前交付证据、覆盖矩阵和剩余外部发布验证以 [F-SWQT-0003](../../.sw/workflow/feature/F-SWQT-0003-研究报告目标态验收闭环.json) 及项目层体验文档为准。

| 维度 | 当前实现事实 | 目标契约 | 差异 |
| --- | --- | --- | --- |
| 主题 API | `Theme` 已含 `light | dark` | 公开 `system | light | dark` 偏好模型 | 类型、运行时和文档未对齐。 |
| 主题运行时 | `ThemeProvider` 固定浅色，setter 为 no-op，`ThemeControl` 不渲染 | 可计算、持久化、同步并实际切换有效主题 | 现有 API 暗示的能力未交付。 |
| CSS 主题 | 已有 `.dark` token 覆盖与 dark variant | 两主题均为经验证的正式能力 | 手工挂载可绕过当前产品约束。 |
| 尺寸系统 | 4px 节奏和 `h-*` 类散布在组件中 | 基础刻度、语义别名、控件尺寸、容器与断点契约 | 缺少单一真源、命名与校验。 |
| 验证 | 未发现主题矩阵、自动 a11y、视觉回归或发布态消费测试 | 质量门禁成为合并/发布条件 | 无法系统识别主题和尺寸漂移。 |

### 3.3. `820/821px` 的证据与定位

`WideScreenGate` 使用 `>=821px` 显示内容、`<=820px` 显示占位提示，两个相邻边界确保不存在 1px 的空档。其服务对象是排序、筛选、选择等无法在窄屏可靠保留的信息密集表格。

仓库历史中未找到“为何精确取 820px”的测量、可用性测试或设计决策记录；该值自初始导入即存在。因此它应被保留为待验证的 `data-dense` 经验阈值，而非设备分类或全局页面断点。Feature 必须以代表性表格的列宽、控件宽度和留白验证该门槛是否仍然合适。

## 4. 研究依据与治理原则

### 4.1. 行业实践

| 实践 | 可迁移原则 | 对本项目的约束 |
| --- | --- | --- |
| DTCG token 模型 | 基础值、语义、组件语义分层；引用与弃用可治理 | 组件只消费语义 token；新增 token 必须说明用途、owner 和迁移路径。 |
| Tailwind v4 主题变量 | 用 `@theme` 以命名变量生成稳定 utility 与响应式 variant | 断点、间距/尺寸、容器统一由 token 包暴露。 |
| WCAG 2.2 | 正式主题均要验证对比度、焦点和非颜色信息传达 | 深浅主题都必须满足 AA；自动检查不能替代人工审查。 |
| Storybook 测试模型 | story 可作为组件状态和视觉基线 | 核心组件须按主题、状态、关键视口拥有可回归基线。 |

### 4.2. 目标分层模型

```text
基础 token（色阶、尺寸、字体）
        ↓ alias / 引用
语义 token（surface、content、action、border、focus、status、space）
        ↓ 主题覆盖或稳定值
组件 token（仅在必要时）
        ↓
组件样式与 Tailwind utilities
```

### 4.3. 核心治理原则

1. 主题改变 token 值，不为同一组件建立平行实现。
2. 尺寸刻度稳定且可组合；只在明确的密度、容器或控件语义下变化。
3. 公开 API、DOM 属性、CSS、文档和测试必须描述同一能力。
4. 例外可以存在，但必须登记原因、语义映射、责任人和验证方式。
5. 质量门禁验证真实渲染与发布态消费，而不只验证 token 文件存在。
6. **目标态优先**：本报告中的工作一律按长期可维护的最佳实践目标态交付；不得以临时开关、no-op API、未验证的预留主题、一次性脚本或“发布后补齐”的质量债替代正式能力。确有迁移措施时，必须记录兼容边界、移除条件、责任人和验收证据。

## 5. 已确认的产品与设计决策

### 5.1. 主题运行时决策

| 决策面 | 已确认选择 |
| --- | --- |
| 正式主题 | 支持浅色与深色两套视觉主题。 |
| 默认策略 | 用户未选择时跟随操作系统的 `prefers-color-scheme`。 |
| 用户覆盖 | 用户可显式选择浅色或深色；覆盖优先于系统偏好，并存入 `localStorage`。 |
| 恢复默认 | ThemeControl 提供“跟随系统”，清除本地覆盖并恢复系统监听。 |
| 作用范围 | 主题全局作用于应用根和全部适用控件；不支持子树级主题隔离。 |
| 控件归属 | `@swui/ui` 提供标准 ThemeControl，以小型下拉菜单呈现三种偏好。 |
| 跨标签页 | 通过 `storage` 事件同步用户覆盖；仅 system 模式响应系统偏好变化。 |
| 首屏体验 | 首次可见渲染必须直接使用有效主题，不得出现明显浅/暗闪烁。 |
| 发布策略 | ThemeControl 的真实启用可作为下一个主版本的破坏性变更发布。 |
| 视觉迁移 | 保持现有浅色视觉值，除 WCAG AA 或明确缺陷外不调整；现有 `.dark` 值仅作为深色起点。 |

### 5.2. 尺寸系统决策

| 决策面 | 已确认选择 |
| --- | --- |
| 系统定位 | 尺寸系统是主题系统的同级基础层，默认跨明暗主题稳定。 |
| 全局断点 | `sm=640px`、`md=768px`、`lg=1024px`、`xl=1280px`、`2xl=1536px`；描述布局空间，不描述设备型号。 |
| 数据密度阈值 | 保留 `820/821px` 为 `data-dense` 专用阈值，不作为全局响应式或页面布局契约。 |
| 间距刻度 | `0, 2, 4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96px`；2px 仅用于边框或视觉微调。 |
| 控件高度 | `compact=32px`（高密度或非主要触控）、`sm=36px`、`md=40px`（默认）、`lg=48px`。 |
| 控件一致性 | Button、Input、Select 和图标按钮共享尺寸契约；图标按钮宽高相等。 |
| 容器尺度 | 提供阅读 `720px`、表单 `560px`、标准内容 `1200px`、宽内容 `1280px` 的 token 与 primitive。 |
| 语义间距 | 提供 `space-control-x`、`space-control-y`、`space-stack`、`space-section`、`space-panel` 等别名。 |

### 5.3. 质量决策

两套正式主题都必须达到 WCAG AA，并通过核心组件自动 a11y、视觉回归和发布产物消费测试。视觉回归至少覆盖紧凑视口、`data-dense` 临界点、`lg` 和 `xl`。

## 6. 目标技术契约

### 6.1. Token 与 Tailwind 契约

`@swui/ui-tokens` 是颜色和尺寸的真源。使用 Tailwind v4 `@theme` 管理 `--breakpoint-*`、`--spacing-*`、`--container-*` 等变量，并维持语义颜色变量与 utility 名的稳定映射。

共享组件优先消费语义 token；基础刻度仅用于已定义的通用尺寸变体。不得在组件内无约束新增裸色值、任意尺寸值、平行 `--color-*` 层或绕过主题选择器的 `dark:` 分支。

### 6.2. 主题运行时契约

1. `ThemeProvider` 接收并暴露 `system | light | dark` 偏好，计算实际的 `light | dark` 有效主题。
2. 显式 `light`/`dark` 覆盖持久化；选择 `system` 时删除覆盖而非存储第三套主题值。
3. 监听系统颜色偏好和浏览器 `storage` 事件；前者仅在 system 模式下生效。
4. 在首屏可见前设置主题选择器和 `color-scheme`，并对存储不可用、禁用 JavaScript 等情况定义安全降级。
5. `ThemeControl` 只暴露真实支持的能力，并遵循全局作用域。

### 6.3. 组件尺寸契约

组件 API 应在适用时公开受限的 `size` 变体，内部高度、横向内边距、图标大小和菜单项高度由同一尺寸层映射。组件不得自行创造同义尺寸层；高密度模式必须显式说明可用范围，不能降低主要触控控件的可用性。

### 6.4. 响应式与容器契约

全局断点用于通用响应式布局。`data-dense` 仅用于信息密集且不能可靠重排的控件；可变为堆叠、卡片或表单的内容必须适配而非 gate。组件库可提供容器 token 与 primitive，但不拥有消费应用的页面布局结构。

## 7. 缺口、治理控制与验收

### 7.1. 缺口清单

| 优先级 | 缺口 | 控制措施 |
| --- | --- | --- |
| 高 | 主题类型、运行时、CSS 与文档不一致 | 在同一变更中交付真实 runtime、主题选择器、ThemeControl 与文档。 |
| 高 | 缺少主题/状态/a11y/视觉/发布态门禁 | 建立核心组件矩阵，并将其接入合并和发布检查。 |
| 高 | 现有 `.dark` 可绕过未支持的运行时 | 将其纳入正式双主题契约，禁止继续以手工类名作为接入方式。 |
| 中 | 尺寸惯例未 token 化 | 建立基础刻度、语义别名、控件尺寸和容器 token，并迁移核心组件。 |
| 中 | 图表含原始颜色与动态变量例外 | 建立 allowlist，记录语义映射与两主题下的对比度验收。 |
| 中 | 仅工作区示例证明消费端接入 | 以 package tarball 在至少两种消费布局中验证 `tokens.css`、`@source` 和组件渲染。 |
| 低 | 没有覆盖率基线 | 建立“核心组件 × 状态 × 主题 × 视口”矩阵和例外统计。 |

### 7.2. 合并与发布门禁

1. **静态门禁**：阻止未批准裸色值、裸尺寸值、未定义 token、循环 alias 和绕过主题选择器的分支。
2. **token 门禁**：解析每个正式主题，校验必填语义 token、引用关系、弃用引用和前景/背景配对。
3. **组件门禁**：核心组件具备 default、hover、focus-visible、disabled、loading、error、empty、selected 等适用状态。
4. **无障碍门禁**：运行自动 WCAG 规则，并为颜色语义、焦点、叠层、图表和动态状态保留人工审查。
5. **视觉门禁**：在两主题、四类关键宽度和稳定夹具下比较截图；差异必须被明确接受或修复。
6. **消费端门禁**：在发布产物而非仅工作区源码中验证 CSS 导入、Tailwind 扫描和组件渲染。
7. **浏览器验收门禁**：必须使用宿主内置浏览器完成真实渲染、主题切换、系统跟随、跨标签同步、关键视口和代表性交互的端到端验收；仅构建、单元测试或截图结果均不能替代此项。

## 8. Feature 建议拆分与验收出口

### 8.1. Feature 目标陈述

为 `@swui/ui-tokens` 与 `@swui/ui` 交付正式的双主题和尺寸系统，使全局主题偏好、语义 token、组件尺寸、响应式密度策略及质量门禁成为一致、可测试、可发布的公共契约。

### 8.2. 建议工作包

本专项以**一个 Feature**交付主题运行时、尺寸系统和全部质量门禁；不得将门禁作为可在正式发布后补齐的独立范围。Feature 内部按依赖关系拆分为细粒度 Step，每个 Step 只在其前置证据已满足后启动。

| 工作包 | 主要产出 | 依赖 | 完成判定 |
| --- | --- | --- | --- |
| A. 契约与 token 基线 | token 清单、尺寸 token、例外清单、公开 API 文档 | 无 | 命名、用途、消费者和迁移策略可审计。 |
| B. 主题运行时 | ThemeProvider、首屏初始化、ThemeControl、持久化与跨标签同步 | A | system/light/dark 行为符合第 5.1 节。 |
| C. 组件尺寸迁移 | 核心控件 size 变体、间距/容器应用、data-dense 复核 | A | 核心控件满足第 5.2 节且无未登记尺寸漂移。 |
| D. 双主题视觉与无障碍 | light/dark token 定稿、状态故事、a11y 与视觉矩阵 | A、B、C | 两主题达到质量决策和第 7.2 节门禁。 |
| E. 发布态消费验证 | tarball 消费夹具、两种安装/hoisting 布局验证 | A、B、C、D | 发布产物可生成完整 CSS 并正确运行。 |

### 8.3. 建议实施顺序

建议在 Feature 设计阶段细化为以下依赖序列：

1. **S1：契约与基线盘点**——完成 token、组件状态、尺寸值、例外和消费者基线；这是所有后续 Step 的证据前提。
2. **S2：token 与尺寸基础**——建立颜色/尺寸 token 分层、断点、容器和语义间距，并完成核心控件尺寸契约。
3. **S3：主题运行时**——在 S1/S2 的契约上实现 ThemeProvider、首屏初始化、持久化、系统监听、跨标签同步和 ThemeControl。
4. **S4：组件双主题迁移**——在 S2/S3 可用后，为核心组件和已登记例外完成 light/dark 语义映射及 `data-dense` 阈值验证。
5. **S5：质量门禁**——在可稳定渲染的组件状态上建立 a11y、视觉回归和静态校验矩阵。
6. **S6：发布态消费者验证**——使用 tarball 在两种消费者布局验证接入；仅在 S1 至 S5 均通过后作为发布出口。

S2 与 S3 可在共享契约确定后并行开展；S4 依赖二者，S5 依赖 S4 的稳定渲染，S6 依赖全部前序 Step。不得在仅完成 CSS `.dark` 覆盖时宣称已支持深色主题。

### 8.4. 目标态交付纪律

1. 每个 Step 的完成产物必须可作为长期公开契约、可维护实现或可重复质量门禁使用。
2. 不得以临时 feature flag、空实现、仅供演示的分支、手工绕过或未验证的“后续补齐”项作为 Feature 完成依据。
3. 如因升级迁移需要短期兼容层，必须在同一 Feature 中定义其使用范围、移除条件、责任人和自动化验证；没有退出计划的兼容层不予接受。
4. 最终发布前必须移除或关闭所有仅为开发过程存在的临时路径，并由浏览器验收和发布态消费者验证证明目标态成立。

### 8.5. Feature 级验收标准

1. 所有第 5 章已确认决策均有代码、文档和自动化证据。
2. 所有第 7.2 节门禁在 CI 或等价可重复环境中通过。
3. 图表和第三方样式例外均已登记并完成双主题审查。
4. `data-dense` 阈值已通过代表性表格的宽度验证；若不符合，修改阈值及文档并更新视觉基线。
5. 完成至少两种发布态消费者布局的打包安装与渲染证明。
6. 使用内置浏览器完成第 7.2 节的浏览器验收门禁，并保存可追溯的验收记录。

### 8.6. 完成后的项目经验沉淀

Feature 完成后，必须在**项目层**而非 SWS 插件方法论层沉淀可复用经验，并至少包含：

1. 主题初始化、持久化、系统偏好和跨标签同步的最终模式与常见陷阱。
2. 尺寸 token、控件尺寸、断点与 `data-dense` 阈值的使用边界和反例。
3. 浏览器验收路径、关键断言、测试夹具和已知环境限制。
4. 视觉/a11y/发布态消费者门禁的运行方式、失败排查路径和例外登记规则。
5. 可复用的消费端接入示例及版本升级注意事项。

该沉淀应写入项目拥有的 `docs/` 文档并由索引链接；不得只留在 PR、聊天记录、Feature 状态或插件资产中。完成交付的最终验收须确认该文档存在、可发现且与实际实现一致。

## 9. 风险与待定的技术选择

### 9.1. 主要风险与缓解

| 风险 | 缓解措施 |
| --- | --- |
| 深色主题视觉质量不足 | 以现有 `.dark` 值为起点，不视为定稿；设计评审、对比度与视觉基线共同验收。 |
| 首屏主题闪烁 | 在应用首次绘制前执行最小主题初始化，并为无 JS 情形定义默认安全值。 |
| 静态规则误伤第三方库或图表 | 使用受审计 allowlist，不以全面豁免替代语义映射。 |
| 视觉快照噪声 | 固定字体、视口、数据夹具、动画和时间相关内容。 |
| `@source` 在外部安装布局失效 | 使用 tarball 与两类消费者布局作为发布门禁。 |
| token 过度扩张 | 新增 token 必须证明不能复用现有语义，并记录 owner 与迁移计划。 |

### 9.2. 不阻塞 Feature 立项的技术选择

以下事项需要在设计阶段收口，但不改变已确认的产品范围：主题选择器与预渲染脚本的具体形式、`localStorage` key 命名、故事/测试工具组合、发布态消费者夹具的具体框架，以及 `data-dense` 阈值验证后的最终数值。
